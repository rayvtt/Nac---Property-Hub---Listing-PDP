## Major Issues Diagnosed & Fixed

### 1. Spine line disappearing on resize (PR #97)

**Root cause:** `grid-template-columns:1fr 140px 1fr` — a `1fr` column defaults to `min-content` minimum width. At intermediate viewport widths the content inside the columns was wider than `1fr` allowed, forcing the grid to overflow and pushing the spine column out of view.

**Fix:** `minmax(0,1fr) 140px minmax(0,1fr)` — `minmax(0,1fr)` allows the column to shrink below its content width, keeping the layout stable at any viewport size.

Also: `.nac-spine-line` was using `left:0; right:0; margin:0 auto` for centering. Replaced with `left:50%; transform:translateX(-50%)` which is more reliable at sub-pixel/zoom levels.

Also added: `.nac-spine-content, .nac-spine-left, .nac-spine-right { min-width:0; }` to allow flex/grid children to shrink.

### 2. Gold highlight text showing as [bracket] on live WP listings (PRs #98 → #102)

**Symptom:** The preview listing (served from GitHub Pages / local file) showed gold italic highlighted words in the opening statement. The live WordPress listing showed `[hội tụ hiếm có]` as plain bracketed text.

**Root cause chain:**
1. WordPress treats `[word]` as shortcode syntax and strips/escapes it when saving ACF fields.
2. The documented fix was to use HTML entities: `&#91;word&#93;` in the `data-stmt` attribute.
3. **But:** `scripts/sync-notion.mjs` loads HTML with `cheerio.load(html, { decodeEntities: false })`. This flag preserves entities in text content but **decodes entity sequences inside attribute values when re-serializing**. So every Notion sync rewrote `&#91;` → `[` in the `data-stmt` attribute, and the next WP sync pushed `[word]` to WordPress, which stripped it.

**Permanent fix (PR #102):** Changed delimiter from `[word]` to `«word»` (French guillemet quotes) in all `data-stmt` attributes across all 4 live PDPs and the template. WordPress has no shortcode syntax for `«»` — no escaping, no entity encoding, nothing for cheerio to corrupt. The JS statement parser regex was updated from `/(\[[^\]]+\])/g` to `/(«[^»]+»)/g`.

**Files changed:** `nobu-da-nang.html`, `mandarin-oriental-residences-da-nang.html`, `pullman-panama-city.html`, `w-suite-istanbul.html`, `_template-listing-pdp.html`, `CLAUDE.md`.

### 3. Spine line z-index cutting through section numbers and boxes (PR #101)

**Symptom:** After PR #99/100 added `z-index:1` to `.nac-spine-line`, the line was rendering on top of section numbers, ROI boxes, and process cards.

**Root cause:** An absolutely positioned element with `z-index:1` is hoisted above all siblings in the same stacking context that have `z-index:auto`. The spine line is the **first child** of `.nac-spine-track`. Siblings (section content, numbers, boxes) painted after it in DOM order, meaning they naturally render above it — but `z-index:1` overrode that ordering.

**Fix:** Removed `z-index:1`. DOM order already handles stacking correctly — the spine line is first in the DOM, sibling sections paint over it naturally.

Also dialed opacity back: `.8` was too heavy visually. Settled on `.5` (gradient endpoints `rgba(196,146,60,.5)`).

### 4. CTA button text appearing light in dark mode (PR #102)

**Symptom:** The "Schedule Consultation" gold pill button on the last image section had its text appearing non-black in dark mode.

**Root cause:** `.nac-pdp[data-theme="dark"]` sets `color:var(--text)` (`#d8d2c5`, a light beige) on the root element. While the button's own `.nac-cine-asp-btn { color:#000 }` declaration should override inherited color, the dark theme selector has higher specificity (`0,2,0` — one class + one attribute) than the button rule (`0,1,0` — one class). On some cascade paths this could allow the inherited light color to bleed through.

**Fix:** Added explicit `.nac-pdp[data-theme="dark"] .nac-cine-asp-btn { color:#000; }` to guarantee black text in dark mode with unambiguous specificity (`0,3,0`).

## data-stmt Delimiter Convention

Opening statement gold highlights use `«word»` (guillemet quotes):

```html
<p data-stmt="Brand City. A «highlighted phrase» and the rest of the sentence."></p>
```

JS parser (in every PDP file):
```js
raw.split(/(«[^»]+»)/g).forEach(function(part){
  var hl = part.charAt(0) === '«' && part.charAt(part.length-1) === '»';
  var text = hl ? part.slice(1, -1) : part;
  // renders .word.hl spans for gold italic animation
});
```

**Never use `[word]`** — WordPress strips it as a shortcode.
**Never use `&#91;word&#93;`** — cheerio `decodeEntities:false` converts it back to `[word]` on every sync.
